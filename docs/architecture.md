# Architecture

## Purpose

`hz-openapi-gen` generates CloudWeGo Hertz routers, handler stubs, model structs, an optional `main.go`, and an optional `generate.go` directly from an OpenAPI 3.x document. OpenAPI is the source of truth; no intermediate IDL or code generator (hz, thriftgo, protoc) is involved.

## Pipeline

`load → extract → render → write`

### 1. Load (`internal/openapi`)

`openapi.Load` parses raw YAML or JSON bytes through `libopenapi.NewDocument` and `BuildV3Model`. Swagger 2.0 documents are rejected at this stage. The caller receives a resolved `*v3.Document` with all `$ref` pointers resolved.

### 2. Extract (`internal/openapi`)

`openapi.Extract` walks the resolved document and produces an `openapi.SpecModel`:

- Paths are iterated in source order; `components.schemas` in source order.
- `operationId` becomes the exported Go handler name (`FuncName`) via `openapi.Exported`, preserving known initialisms (e.g. `getUserByID` → `GetUserByID`).
- OpenAPI path templates are rewritten to Hertz syntax (`{id}` → `:id`) by `openapi.hertzPath`.
- Schema `$ref` pointers are resolved to Go type names; inline and unsupported constructs (`oneOf`/`anyOf`) degrade to `interface{}` with diagnostics.
- Diagnostics (`diag.Set`) collect blocking errors and non-blocking warnings, reported in a stable deterministic order.

All Go names, types, and Hertz paths are fully computed during extraction. The generator package never touches `libopenapi`.

### 3. Render (`internal/generator`)

`generator.Generate` produces `[]generator.GeneratedFile` in this fixed order:

| # | Kind | Mode | File(s) |
|---|------|------|---------|
| 1 | router | `WriteCover` | `biz/router/register.go` |
| 2 | opid | `WriteCover` | `biz/opid/opid.go` (delegate mode only) |
| 3 | handler | `WriteMerge` (stub) / `WriteCover` (delegate) | `biz/handler/{tag}.go` (one per tag) |
| 4 | model | `WriteCover` | `biz/model/{schema}.go` (one per schema) |
| 5 | main | `WriteCreate` | `main.go` (optional, `-generate-main`) |
| 6 | generate | `WriteCreate` | `generate.go` (default, `-generate-go-generate`) |

In delegate mode, handlers render from the `delegateHandler` template and write as `WriteCover` with `DoNotEditMarker` (fully owned, replaced on regen). In the default stub mode they render from `handler`/`handlerFunc` and write as `WriteMerge` with `GeneratedMarker` (extension points).

Templates are defined as named Go constants in `templates.go` and rendered with `text/template`. Complete files go through `render`, which applies `hzutil.FormatGo` (`go/format.Source`) with a hard failure on invalid Go. Standalone handler functions go through `renderHandlerFunc`, which returns raw function snippets — these are never formatted as standalone Go files because `go/format.Source` requires a complete file.

### 4. Write (`internal/writer`)

`writer.Write` iterates the generated file plan and applies each file's write mode. All-or-nothing: if any file fails marker checks, parsing, import insertion, or formatting, zero files are written.

## Data model

`openapi.SpecModel` is the single data structure shared between extraction and generation:

| Type | Purpose |
|------|---------|
| `SpecModel` | Top-level container: `Title`, `Version`, `Operations []OpDef`, `Models []ModelDef` |
| `OpDef` | One HTTP operation: `FuncName`, `Method`, `HertzPath`, `RequestModel`, `ResponseModel`, `Parameters []ParamDef`, `Responses []ResponseDef` |
| `ModelDef` | One Go struct: `Name`, `PackageName`, `Fields []FieldDef`, `UsesTime() bool` |
| `FieldDef` | One struct field: `Name` (exported Go), `JSONName`, `GoType`, `Required`, `ValidationTags` |
| `ParamDef` | Path/query/header/cookie parameter: `GoName`, `In`, `GoType`, `BindTag` |
| `ResponseDef` | Declared response: `StatusCode`, `Description`, `ModelName`, `MediaType` |

Name computation (`openapi.Exported`, `openapi.FileStem`, `openapi.refName`) is deterministic and stateless. The `UsesTime()` method on `ModelDef` avoids an unconditional `"time"` import in model files.

## Generated file modes

| Mode | Applies to | Marker required | Behavior on existing file |
|------|-----------|-----------------|--------------------------|
| `WriteCover` | Router, models, opid constants, delegate handlers | `DoNotEditMarker` (or `GeneratedMarker`) | Replace entire content. Identical content is skipped (no write). A file carrying either marker is accepted, so a stub→delegate switch regenerates handler files without global `-force`. |
| `WriteCreate` | `main.go`, `generate.go` | None | Skip if exists; never overwrite. |
| `WriteMerge` | Stub handlers | `GeneratedMarker` | Append only missing handler functions. Duplicate detection is function-name based. Existing custom code is preserved. |

**Markers:**

- `generator.DoNotEditMarker`: `// Code generated by hz-openapi-gen. DO NOT EDIT.`
- `generator.GeneratedMarker`: `// Code generated by hz-openapi-gen.`

Without `-force`: `WriteCover` files without `DoNotEditMarker` or `GeneratedMarker` on the first line are refused. `WriteMerge` files without `GeneratedMarker` on the first line are refused. Both raise `writer.ErrUnsafeOverwrite` (exit code 3).

With `-force`: marker checks are skipped. Handler files are still merged (never overwritten).

## Handler merge semantics

Handler merge is a structured append, not a text diff. Each `generator.GeneratedFile` with `Mode: WriteMerge` carries a `*generator.GoMergePlan`:

- `GoMergePlan` holds `[]GoFunction`.
- Each `GoFunction` has a `Name` (e.g. `GetUserByID`), `Content` (raw function snippet from `renderHandlerFunc`), and `Imports []GoImport`.

`writer.mergeGoFile` processes each function:

1. Skip if `hzutil.HasGoFunction` detects a matching `func Name(` already present (regex-based, not AST).
2. For each import: skip if the import path string is already present in the file; otherwise insert via `hzutil.AddImportForContent`, which parses the file with `go/parser`, adds the import with `astutil.AddNamedImport`, and formats back with `format.Node`.
3. Append the function snippet at the end of the file with a blank-line separator.
4. Format the entire merged file with `hzutil.FormatGo` (`go/format.Source`). Any failure here aborts the entire write.

**Delegate→stub diagnostic.** When stub mode regenerates over handler files last written in delegate mode, each existing function body still carries a `// delegated` marker. Merge keeps those bodies (the functions already exist) but `writer` records a warning per function — `handler <path>: existing function <name> was generated in delegate mode and was not replaced` — collected in `Result.Warnings` and reported by `main`. To regenerate fresh stubs, delete or rename the delegate handler files first.

## Delegate handler mode

Delegate mode (`-handler-mode=delegate`) targets proxy/gateway services where every operation dispatches to a single runtime shim. Three flags configure the delegate target, all required in delegate mode: `-delegate-import` (package import path), `-delegate-package` (package alias/name), and `-delegate-func` (function name). Package and function must be valid Go identifiers, and the package name must not collide with the import names the handler template already uses (`context`, `app`, `consts`, `opid`).

In delegate mode:

- Handlers render from the `delegateHandler` template and write as `WriteCover` with `DoNotEditMarker`. Each handler calls `<package>.<func>(ctx, c, opid.<Operation>)` and is replaced wholesale on regeneration.
- An operationId constants file (`biz/opid/opid.go`, kind `opid`, `WriteCover`, `DoNotEditMarker`) is emitted from the `opidConstants` template — one exported constant per operation, named like the handler func and valued with the verbatim `operationId`. This makes the dispatch key a single source of truth: a spec rename changes the constant and fails to compile in callers.
- Every selected operation must have a non-empty `operationId`, regardless of `-strict`; otherwise generation fails before any file is written.

Safe mode switches: `WriteCover` accepts either `DoNotEditMarker` or `GeneratedMarker`, so a stub→delegate switch regenerates the tool-owned handler files without global `-force`. A delegate→stub switch is the inverse — merge keeps the existing delegate bodies and emits a per-function warning (see [Handler merge semantics](#handler-merge-semantics)).

## Package responsibilities

| Package | Role |
|---------|------|
| `main.go` | CLI entry point: flag parsing, delegate-mode validation, orchestration, exit codes, result reporting |
| `internal/openapi` | OpenAPI loading (`libopenapi`), extraction to `SpecModel`, Go name computation, schema-to-Go-type mapping |
| `internal/generator` | Template definitions, `Generate` orchestration, file plan construction, `GoMergePlan`, handler-mode selection |
| `internal/hzutil` | Go formatting (`go/format.Source`), import insertion (`astutil.AddNamedImport`), function existence check (regex), path construction (`SubPackage`, `SubDir`) |
| `internal/writer` | Write-mode dispatch, marker checks, file comparison, `mergeGoFile`, delegate-body diagnostic, all-or-nothing write |
| `internal/diag` | Diagnostic collection and deterministic reporting |
| `testdata/` | Spec fixtures (`minimal.yaml`, `petstore.yaml`) and golden snapshots (`golden/`, `golden-delegate/`) |

## Safety rules

1. **Router and model files are disposable.** Regeneration replaces them entirely. They must carry `DoNotEditMarker`.
2. **Optional `main.go` and `generate.go` are create-only.** Once present, they are never overwritten by the generator — even with `-force`.
3. **Stub handler files are extension points.** Custom code between generated handler functions is preserved. Only missing functions are appended. Delegate handler files, by contrast, are fully owned and replaced on regeneration.
4. **Unmarked existing files are refused** unless `-force` is set. This prevents the generator from silently overwriting hand-written code.
5. **All-or-nothing writes.** Parse errors, format errors, import insertion errors, and marker violations all abort before any file is written.
6. **Handler file format errors block the entire generation.** If a handler file does not parse as valid Go, the generator refuses to merge and directs the operator to fix the file.
7. **Delegate mode owns handler files.** Switching a service to delegate mode overwrites stub handler bodies because they carry `GeneratedMarker`, which cover mode accepts. This is opt-in and intended; a service carrying real handler logic is not a delegate-mode candidate.
