# Runbook

## Generate a service from a spec

```
go run ./cmd/hz-openapi-gen -spec=api/openapi.yaml -out=. -module=example.com/service
```

Expected output:

```
hz-openapi-gen: loaded api/openapi.yaml
hz-openapi-gen: extracted <N> operations, <M> schemas
hz-openapi-gen: wrote biz/router/register.go
hz-openapi-gen: wrote biz/handler/pet.go
hz-openapi-gen: wrote biz/model/pet.go
hz-openapi-gen: done
```

- `wrote <path>` — new file created.
- `updated <path>` — existing file with correct marker was overwritten (router/models) or merged (handlers).
- `skipped <path> (exists, not generated)` — `WriteCreate` file already present (`main.go`, `generate.go`).
- Exit code `0`.

## Validate without writing

```
go run ./cmd/hz-openapi-gen -spec=api/openapi.yaml -module=example.com/service -validate-only
```

Expected: exit `0` with `validation passed: <N> operations, <M> schemas, <W> warnings`. No files written. Warnings and errors print to stderr. Non-zero exit on errors.

## Preview writes

```
go run ./cmd/hz-openapi-gen -spec=api/openapi.yaml -out=. -module=example.com/service -dry-run
```

Expected: no files written. Output shows `hz-openapi-gen: dry-run (no files written)` followed by `planned files:` and result lines:

| Prefix | Meaning |
|--------|---------|
| `+` | File will be created |
| `~` | File will be changed (overwritten or merged) |
| `=` | File exists and has identical content (no-op) |
| `(skip)` | `WriteCreate` file already exists (skipped) |

## Regenerate safely after editing handlers

Handler files (`biz/handler/*.go`) are extension points. Custom code between generated functions is preserved across regenerations.

**Safety mechanism:** The generator appends only handler functions that are not already present. Detection uses function-name matching (`hzutil.HasGoFunction`).

**Required marker:** The first line of the handler file must contain `// Code generated by hz-openapi-gen.` (without `DO NOT EDIT`). If the marker is missing and `-force` is not set, the generator refuses to merge (exit code 3).

**Workflow:**

1. Edit the handler file — replace the `501 Not Implemented` stub with real logic.
2. Add new endpoints to the OpenAPI spec.
3. Rerun the generator. Existing handler functions are left untouched; new ones are appended.
4. Optional: delete a handler function from the Go file and rerun to regenerate it fresh.

## Resolve unsafe overwrite errors

Exit code `3` and the error message `unsafe overwrite prevented: <path> does not contain "<marker>"` means a file the generator wants to write to lacks the required generated-code marker.

| File type | Required marker |
|-----------|----------------|
| Router (`biz/router/register.go`) | `// Code generated by hz-openapi-gen. DO NOT EDIT.` |
| Models (`biz/model/*.go`) | `// Code generated by hz-openapi-gen. DO NOT EDIT.` |
| Handlers (`biz/handler/*.go`) | `// Code generated by hz-openapi-gen.` |
| `main.go` | None (skipped if exists) |
| `generate.go` | None (skipped if exists) |

**Resolution options (in order of preference):**

1. **The file was originally generated but the marker was removed.** Restore the marker on the first line and rerun.
2. **The file is hand-written and should stay that way.** Move it or change `-package-base` to write generated files to a different directory.
3. **You are certain the generator should overwrite/merge anyway.** Use `-force`. This skips marker checks but preserves handler merge semantics — handler files are still merged, not overwritten.

Do not use `-force` as a permanent workaround. The markers exist to prevent accidental data loss.

## Resolve handler merge errors

Handler merge can fail for several reasons:

### Parse errors

```
merge biz/handler/pet.go: add import "…" for GetPet: parse: <details>
```

The existing handler file does not parse as valid Go. Fix the file so it compiles, then rerun.

### Import insertion errors

```
merge biz/handler/pet.go: add import "…" for GetPet: add import "…": already present or failed
```

An import that should be added could not be inserted. Either the import is already present (unexpected path mismatch) or the AST is malformed. Verify the import paths and fix the file.

### Format errors

```
merge biz/handler/pet.go: <format error>
```

The merged result does not format with `go/format.Source`. This typically means the appended function snippet introduces syntax errors relative to the existing file. Fix the handler file so it is well-formed and rerun.

### Duplicate function names

Not an error — the generator silently skips functions already present. If a function is skipped but you expected it to be regenerated, delete the old function body and rerun.

## Regenerate goldens

```
make golden
go test ./internal/generator/...
```

Golden snapshots are generated from `testdata/minimal.yaml`. Run after any template or name-computation change.

## Verify generated services compile

```
make test-generated
```

This test generates real services from `testdata/minimal.yaml` and `testdata/petstore.yaml` into temp directories, runs `go mod tidy`, `go build ./...`, and `go vet ./...` against each. Requires network access (module downloads).

Skipped with `-short`. Flaky behavior usually indicates:
- Missing or outdated `go.sum` entries in the project.
- Network issues during `go mod tidy`.
- `libopenapi` version mismatch in the generated `go.mod`.


## Resolve quality gate failures

- `golangci-lint run ./cmd/... ./internal/...` — fix reported code issues; suppress only confirmed false positives with `//nolint:<linter> // <specific reason>`.
- `go tool govulncheck ./cmd/... ./internal/...` — upgrade the vulnerable module to the first fixed version, run `go mod tidy`, and rerun the scan. Note: govulncheck currently panics on Go 1.26 due to an `x/tools` bug; the CI job is set to `allow_failure` until the upstream fix lands.
- `go test ./cmd/hz-openapi-gen -run TestGeneratedServiceCompiles` — rerun when generated-service compile fails; inspect the temp service error in test output.
## Exit codes

| Code | Meaning |
|------|---------|
| `0` | Success |
| `1` | Parse, validation, or generation error — check stderr for diagnostics |
| `2` | Usage error — missing required flags, invalid flag values |
| `3` | Unsafe overwrite prevented — file lacks required generated-code marker |

## Check version

```
hz-openapi-gen -version
```

Prints the build version (set at compile time via LDFLAGS). Development builds show `dev`; tagged releases show the git tag.
