# Runbook

## Generate a service from a spec

```
go run . -spec=api/openapi.yaml -out=. -module=example.com/service
```

Expected output:

```
hz-openapi-gen: loaded api/openapi.yaml
hz-openapi-gen: extracted <N> operations, <M> schemas
hz-openapi-gen: wrote biz/router/register.go
hz-openapi-gen: wrote biz/handler/pet.go
hz-openapi-gen: wrote biz/model/pet.go
hz-openapi-gen: done
```

- `wrote <path>` — new file created.
- `updated <path>` — existing file with correct marker was overwritten (router/models) or merged (handlers).
- `skipped <path> (exists, not generated)` — `WriteCreate` file already present (`main.go`, `generate.go`).
- Exit code `0`.

## Validate without writing

```
go run . -spec=api/openapi.yaml -module=example.com/service -validate-only
```

Expected: exit `0` with `validation passed: <N> operations, <M> schemas, <W> warnings`. No files written. Warnings and errors print to stderr. Non-zero exit on errors.

## Preview writes

```
go run . -spec=api/openapi.yaml -out=. -module=example.com/service -dry-run
```

Expected: no files written. Output shows `hz-openapi-gen: dry-run (no files written)` followed by `planned files:` and result lines:

| Prefix | Meaning |
|--------|---------|
| `+` | File will be created |
| `~` | File will be changed (overwritten or merged) |
| `=` | File exists and has identical content (no-op) |
| `(skip)` | `WriteCreate` file already exists (skipped) |

## Regenerate safely after editing handlers

Handler files (`biz/handler/*.go`) are extension points. Custom code between generated functions is preserved across regenerations.

**Safety mechanism:** The generator appends only handler functions that are not already present. Detection uses function-name matching (`hzutil.HasGoFunction`).

**Required marker:** The first line of the handler file must contain `// Code generated by hz-openapi-gen.` (without `DO NOT EDIT`). If the marker is missing and `-force` is not set, the generator refuses to merge (exit code 3).

**Workflow:**

1. Edit the handler file — replace the `501 Not Implemented` stub with real logic.
2. Add new endpoints to the OpenAPI spec.
3. Rerun the generator. Existing handler functions are left untouched; new ones are appended.
4. Optional: delete a handler function from the Go file and rerun to regenerate it fresh.

## Switch to delegate handler mode

Delegate mode generates fully owned handler files that call a single delegate function, plus the operationId constants file. It is intended for proxy/gateway services that own none of their handler bodies:

```
go run . -spec=api/openapi.yaml -out=. -module=example.com/service \
  -handler-mode=delegate \
  -delegate-import=example.com/service/internal/public \
  -delegate-package=public \
  -delegate-func=Serve
```

Handler files switch to cover mode and carry the DO-NOT-EDIT marker, so each regeneration replaces them wholesale. `biz/opid/opid.go` is generated with one exported constant per operation (named like the handler func, valued with the verbatim `operationId`).

**Warning: hand-edited stub handlers are overwritten.** Stub handler files carry the shorter generated marker (`// Code generated by hz-openapi-gen.`), which cover mode accepts for overwrite. Any business logic you added to a stub is lost when you switch to delegate mode — without `-force`. Delegate mode is for services that own none of their handler bodies; if a stub carries real logic, it is not a delegate-mode candidate.

**Required in delegate mode:** non-empty `-delegate-import` and `-delegate-func`; `-delegate-package` and `-delegate-func` must be valid Go identifiers; `-delegate-package` must not collide with `context`, `app`, `consts`, or `opid`; every operation must have a non-empty `operationId` (enforced regardless of `-strict`).

## Switch back from delegate to stub handlers

Running in stub mode (`-handler-mode=stub`, the default, or omitting the flag) switches handler files back to merge mode. Existing delegate bodies are kept — the functions already exist, so merge does not append — but the writer emits a warning per function:

```
hz-openapi-gen: warning: handler biz/handler/<tag>.go: existing function <Name> was generated in delegate mode and was not replaced
```

The delegate handler files still carry the generated marker (the DO-NOT-EDIT line contains it), so stub-mode merge accepts them and only appends any missing functions.

To regenerate fresh `501` stub bodies, delete or rename the delegate handler files first, then run in stub mode. The generator recreates them as extension-point stubs.

## Resolve unsafe overwrite errors

Exit code `3` and the error message `unsafe overwrite prevented: <path> does not contain "<marker>"` means a file the generator wants to write to lacks the required generated-code marker.

| File type | Required marker |
|-----------|----------------|
| Router (`biz/router/register.go`) | `// Code generated by hz-openapi-gen. DO NOT EDIT.` |
| Models (`biz/model/*.go`) | `// Code generated by hz-openapi-gen. DO NOT EDIT.` |
| Handlers (`biz/handler/*.go`) | `// Code generated by hz-openapi-gen.` |
| `main.go` | None (skipped if exists) |
| `generate.go` | None (skipped if exists) |

**Resolution options (in order of preference):**

1. **The file was originally generated but the marker was removed.** Restore the marker on the first line and rerun.
2. **The file is hand-written and should stay that way.** Move it or change `-package-base` to write generated files to a different directory.
3. **You are certain the generator should overwrite/merge anyway.** Use `-force`. This skips marker checks but preserves handler merge semantics — handler files are still merged, not overwritten.

Note: cover mode (router/models/opid/delegate handlers) accepts either the DO-NOT-EDIT or the shorter generated marker, so a stub→delegate switch regenerates handler files without `-force`. Do not use `-force` as a permanent workaround. The markers exist to prevent accidental data loss.

## Resolve handler merge errors

Handler merge can fail for several reasons:

### Parse errors

```
merge biz/handler/pet.go: add import "…" for GetPet: parse: <details>
```

The existing handler file does not parse as valid Go. Fix the file so it compiles, then rerun.

### Import insertion errors

```
merge biz/handler/pet.go: add import "…" for GetPet: add import "…": already present or failed
```

An import that should be added could not be inserted. Either the import is already present (unexpected path mismatch) or the AST is malformed. Verify the import paths and fix the file.

### Format errors

```
merge biz/handler/pet.go: <format error>
```

The merged result does not format with `go/format.Source`. This typically means the appended function snippet introduces syntax errors relative to the existing file. Fix the handler file so it is well-formed and rerun.

### Duplicate function names

Not an error — the generator silently skips functions already present. If a function is skipped but you expected it to be regenerated, delete the old function body and rerun.

## Resolve delegate handler mode issues

### Delegate→stub warning

```
hz-openapi-gen: warning: handler biz/handler/<tag>.go: existing function <Name> was generated in delegate mode and was not replaced
```

You switched a service from delegate back to stub mode. Merge keeps the existing delegate bodies (the functions already exist) and warns instead of silently leaving them. To regenerate fresh `501` stubs, delete or rename the delegate handler files and rerun in stub mode.

### Delegate mode requires operationId

```
hz-openapi-gen: delegate mode requires operationId for operation <method> <path>
```

Every operation must have a non-empty `operationId` in delegate mode — the generated body is `<delegate>(ctx, c, opid.<Operation>)`, which needs a constant name. This check runs regardless of `-strict` and fails (exit code `1`) before any file is written. Add an `operationId` to the operation in the spec.

### Delegate flag validation

Usage errors (exit code `2`) are reported before any file is written:

| Message | Cause |
|---------|-------|
| `-handler-mode must be stub or delegate` | unsupported `-handler-mode` value |
| `-delegate-import is required in delegate mode` | missing `-delegate-import` |
| `-delegate-func is required in delegate mode` | missing `-delegate-func` |
| `-delegate-package must be a valid Go identifier` | `-delegate-package` is not an identifier |
| `-delegate-func must be a valid Go identifier` | `-delegate-func` is not an identifier |
| `-delegate-package "<name>" is reserved` | `-delegate-package` is `context`, `app`, `consts`, or `opid` |

## Regenerate goldens

```
make golden
go test ./internal/generator/...
```

Golden snapshots are generated from `testdata/minimal.yaml`. Delegate mode has a parallel set under `testdata/golden-delegate/`, regenerated the same way (`UPDATE_GOLDEN=1 go test ./internal/generator/...` or `make golden`). Run after any template or name-computation change.

## Verify generated services compile

```
make test-generated
```

This test generates real services from `testdata/minimal.yaml` and `testdata/petstore.yaml` into temp directories, runs `go mod tidy`, `go build ./...`, and `go vet ./...` against each. A `delegate` subtest also generates a delegate-mode service with a tiny local delegate package and the constants file. Requires network access (module downloads).

Skipped with `-short`. Flaky behavior usually indicates:
- Missing or outdated `go.sum` entries in the project.
- Network issues during `go mod tidy`.
- `libopenapi` version mismatch in the generated `go.mod`.


## Resolve quality gate failures

- `golangci-lint run . ./internal/...` — fix reported code issues; suppress only confirmed false positives with `//nolint:<linter> // <specific reason>`.
- `go tool govulncheck . ./internal/...` — upgrade the vulnerable module to the first fixed version, run `go mod tidy`, and rerun the scan. Note: govulncheck currently panics on Go 1.26 due to an `x/tools` bug; the CI job is set to `allow_failure` until the upstream fix lands.
- `go test . -run TestGeneratedServiceCompiles` — rerun when generated-service compile fails; inspect the temp service error in test output.
## Exit codes

| Code | Meaning |
|------|---------|
| `0` | Success |
| `1` | Parse, validation, or generation error — check stderr for diagnostics |
| `2` | Usage error — missing required flags, invalid flag values |
| `3` | Unsafe overwrite prevented — file lacks required generated-code marker |

## Check version

```
hz-openapi-gen -version
```

Prints the build version (set at compile time via LDFLAGS). Development builds show `dev`; tagged releases show the git tag.
